# 生产可用性评估与分阶段任务

## 结论

当前项目已经具备“可演示的业务原型”和一部分真实后端能力，但还不能直接按生产系统上线。

核心原因不是“没有代码”，而是生产系统需要的闭环能力还不完整：真实支付、实名与手机号校验、敏感数据保护、并发一致性、后台运营工具、自动化测试、部署运维、监控告警、数据备份和上线回滚都还没有达到生产要求。

评估日期：2026-06-25
评估范围：根目录静态前端、`backend/` Go/Gin API、`database/` SQL 迁移、现有项目文档。

## 生产可用性缺口

### P0：上线前必须解决

| 缺口 | 当前风险 | 需要达到的生产状态 |
|---|---|---|
| 手机号登录没有短信验证码 | 任意人可用任意手机号创建或登录用户 | 接入短信验证码、频控、验证码过期、错误次数限制 |
| 实名认证直接标记为 `verified` | 无真实实名核验，证件数据可信度为零 | 接入实名校验服务，至少区分待审核、通过、失败 |
| 默认密钥可进入运行态 | `JWT_SECRET`、`PAYMENT_WEBHOOK_SECRET` 有开发默认值 | 启动时校验生产环境密钥，默认值直接拒绝启动 |
| CORS 默认 `*` | 任意来源可调 API，配合 token 泄漏风险高 | 生产必须配置明确域名白名单 |
| 环岛游交易类接口公开 | 锁座、出票、退票、改签等接口未挂用户/后台鉴权 | 查询类公开，交易类按用户或后台权限隔离，并校验订单归属 |
| 支付不是真实闭环 | 当前只有自定义签名回调，没有微信/支付宝下单、验签、退款闭环 | 接入真实支付网关，下单、回调、退款、查单、对账全链路 |
| 票核销并发不安全 | 先查 `valid` 再更新，两个请求可能重复核销 | 使用条件更新 `WHERE id=? AND status='valid'`，并检查 affected rows |
| 司机提现可重复占用余额 | 可提现余额没有扣除 `pending` 提现，可能多次提交超额申请 | pending/approved/transferred 全部占用余额，提现创建必须事务化 |
| 订单锁库存没有过期释放 | 未支付订单长期占用 `locked_stock` | 增加订单超时取消任务，释放锁定库存 |
| 金额使用 float | 金额比较和累计可能有精度误差 | 后端金额统一用分为单位的整数，或 decimal 类型库 |
| 敏感数据明文存储 | 身份证、证件号、收款账号等直接存数据库 | 字段级加密、脱敏输出、密钥轮换方案 |
| 自动化测试缺失 | 关键交易路径没有回归保护 | 至少覆盖认证、下单、支付回调、核销、提现、环岛游状态流转 |

### P1：生产稳定运营必须补齐

| 缺口 | 当前风险 | 需要达到的生产状态 |
|---|---|---|
| 后台缺少产品/SKU/排期管理 | 只能展示已有数据，运营无法完整维护商品和库存 | 后台支持产品、SKU、排期、价格、库存、上下架管理 |
| 退款流程只是标记 | 未调用支付平台退款，未恢复库存、票状态、佣金 | 退款审核、支付退款、库存/票/佣金回滚、退款回调全链路 |
| 发票只是申请记录 | 无开票处理、状态流转和发票文件 | 后台开票、发票号、发票下载/邮件发送 |
| 用户和客服后台缺失 | 无用户封禁、实名审核、投诉处理闭环 | 用户管理、实名审核、工单处理、操作审计 |
| 管理操作审计未落地 | 已有 `audit_logs` 表，但关键操作未统一写日志 | 登录、退款、核销、提现、司机审核、参数变更全部审计 |
| 配置加载频繁且无环境分层 | 中间件和 token TTL 多次 `config.Load()`，生产行为难控 | 启动时加载一次配置，区分 dev/staging/prod |
| GORM 生产日志过重 | 默认 `logger.Info` 可能记录大量 SQL 和敏感字段 | 生产改 warning/error，敏感字段过滤 |
| 错误处理不一致 | 多处事务提交错误未检查或 DB 错误被忽略 | 统一错误处理、事务 helper、明确日志上下文 |
| 环岛游供应商容错不足 | 供应商失败、超时、重试、幂等、对账不完整 | HTTP timeout、重试策略、幂等键、供应商状态对账任务 |
| 前端仍有原型页面数据 | 部分页面保留演示默认值和静态流程 | 用户真实流程页面与纯展示原型分离 |

### P2：规模化和长期维护需要补齐

| 缺口 | 当前风险 | 需要达到的生产状态 |
|---|---|---|
| 没有 CI/CD | 每次发布依赖人工命令 | PR 自动测试、构建、镜像、部署审批 |
| 没有部署清单 | 缺 Dockerfile、systemd、Nginx、环境变量说明 | 可重复部署脚本和回滚方案 |
| 没有监控告警 | 线上故障只能靠用户反馈 | 指标、日志、错误追踪、告警、仪表盘 |
| 没有备份恢复演练 | 数据库故障或误操作恢复不可控 | 定时备份、恢复演练、RPO/RTO 目标 |
| 没有压测和容量规划 | 高峰下库存、支付、核销风险未知 | 压测脚本、容量基线、慢 SQL 治理 |
| 缺 API 文档和契约 | 前后端和第三方联调成本高 | OpenAPI/Apifox 文档、示例请求、错误码规范 |
| 缺隐私合规文档 | 证件、手机号、支付信息处理责任不清 | 隐私政策、数据保留、数据删除和授权留痕 |

## 分阶段任务

本阶段计划按当前目标重排：第一步只做前端工程化重构，用 `uni-app + Vue 3 + TypeScript` 把现有静态 HTML 原型迁移成可持续开发的小程序前端；业务功能补齐、后端加固、真实短信/微信登录/微信支付/供应商正式接入，都放到前端重构完成之后。

### 阶段 0：冻结前端重构边界

目标：明确第一步只解决“小程序前端工程化”和“现有页面迁移”，避免把新业务开发、真实支付和后端大改混入重构阶段。

任务：

1. 确认技术栈固定为 `uni-app + Vue 3 + TypeScript`。
2. 建立页面迁移清单：优先迁移首页、环岛游订票、订单/票券、司机端、我的；后台管理继续保留 Web，不进入小程序重构范围。
3. 标记第一阶段不做的事项：真实微信登录、手机号授权、短信验证码、微信支付、供应商正式账号、退款资金流、提现真实打款、完整后台运营。
4. 梳理现有静态 HTML 中可复用的内容：页面结构、业务文案、视觉风格、交互流程、接口调用字段。
5. 定义重构验收口径：小程序能跑、页面能看、核心跳转成立、Mock 数据可展示、接口层可切换真实后端。

验收标准：

- 第一阶段目标被限定为前端重构，不以真实接入和新增业务完成度作为验收条件。
- 每个要迁移的页面都有来源 HTML、目标小程序路由和对应验收点。
- 后续功能开发依赖项单独进入后续阶段，不阻塞重构开工。

### 阶段 1：搭建 uni-app 小程序工程

目标：建立新的小程序前端工程，把项目从静态 HTML 页面切换到可维护的 Vue 3/TypeScript 结构。

任务：

1. 新建 `miniprogram/` 或等价目录，初始化 `uni-app + Vue 3 + TypeScript` 工程。
2. 配置微信小程序构建目标、基础脚本、环境变量、路径别名和 TypeScript 校验。
3. 建立目录结构：`pages/`、`components/`、`api/`、`stores/`、`types/`、`utils/`、`static/`。
4. 配置基础页面路由和 tabBar：首页、订单、我的；司机端可作为独立入口或分包入口。
5. 封装统一请求层：base URL、请求/响应类型、错误提示、token 注入、Mock/真实 API 切换。
6. 建立基础状态管理：用户登录态、订单草稿、环岛游选择状态、司机登录态。
7. 建立全局样式变量：颜色、字号、间距、按钮、表单、价格、卡片、底部操作栏。

验收标准：

- 小程序项目可在微信开发者工具中编译运行。
- `npm`/`pnpm` 脚本可完成开发构建和类型检查。
- 请求层和 Mock 数据可在不依赖真实接入的情况下驱动页面展示。

### 阶段 2：迁移现有页面为小程序页面

目标：先完成“看起来像、走得通”的前端迁移，不在这一阶段追求完整真实业务闭环。

任务：

1. 迁移首页核心内容：城市文旅入口、产品入口、环岛游入口、订单入口、司机入口。
2. 迁移环岛游订票流程页面：班次选择、乘客填写、订单确认、模拟支付页、出票结果页。
3. 迁移订单/票券页面：订单列表、订单详情、票券卡片、二维码/码内容展示占位。
4. 迁移司机端页面：登录、二维码、佣金明细、钱包、提现申请、提现记录。
5. 迁移“我的”页面：用户信息、常用出行人、订单入口、客服/协议入口。
6. 把现有 HTML 中的浏览器 DOM 操作改为 Vue 组件状态和 uni-app 生命周期。
7. 将本地图片资源迁入小程序静态资源目录，并压缩不适合小程序包体的图片。

验收标准：

- 首批页面在小程序中可打开、可跳转、可返回。
- 页面不再依赖原 HTML 文件、浏览器全局变量或 DOM 查询。
- 主要视觉层级、表单结构、订单状态和票券展示与现有原型保持一致。

### 阶段 3：前端接口适配与 Mock 闭环

目标：让重构后的小程序可以用 Mock 或现有后端数据跑通展示流程，为后续业务开发打底。

任务：

1. 定义前端 API 类型：产品、班次、价格、乘客、订单、票券、司机、提现。
2. 为每类接口提供 Mock 数据，覆盖成功、空数据、失败、过期、库存不足等状态。
3. 对接当前已有后端接口中无需真实接入的部分：产品列表、环岛游查询、订单查询、司机钱包等。
4. 为尚不稳定或未来要替换的接口建立适配层，避免页面直接绑定临时字段。
5. 统一错误态、空态、加载态、重试态和登录失效态。
6. 输出小程序前端接口契约草稿，标明哪些字段来自现有后端，哪些字段是后续需要补齐。

验收标准：

- 小程序可以在 Mock 模式下完整演示首页、环岛游订票、订单/票券、司机端页面。
- 小程序可以切换到本地后端读取已有公开接口数据。
- 后续后端接口变更主要修改 `api/` 适配层，不需要大面积改页面。

### 阶段 4：用户端业务功能开发

目标：在前端工程和页面迁移完成后，再开始补齐用户端真实业务功能。

任务：

1. 接入用户登录态，开发期可继续使用手机号模拟登录，预留微信登录和手机号授权。
2. 完成环岛游选择班次、填写实名乘客、锁票/下单、模拟支付、出票结果、订单查询流程。
3. 补齐订单状态流转：待锁票、待支付、已出票、已取消、已退款、已核销、改签中。
4. 补齐退款、改签、取消锁票等用户入口和状态展示。
5. 对手机号、证件号、票码等敏感内容做前端脱敏展示。
6. 增加表单校验、重复提交保护、锁票倒计时、失败重试和异常提示。

验收标准：

- 使用 Mock 或测试后端可以完整走通一笔环岛游订单。
- 用户端主要异常场景有明确页面反馈。
- 页面主流程不依赖真实微信支付和正式供应商账号。

### 阶段 5：司机端与后台支撑功能开发

目标：在小程序前端稳定后，补齐司机端和必要后台支撑，仍不把完整后台迁入小程序。

任务：

1. 完成司机登录、个人信息、推广二维码、佣金明细、钱包、提现申请、提现记录。
2. 开发期提现只做申请和审核状态，真实打款后置。
3. 补齐后台首版支撑：订单查询、司机审核、提现审核、环岛游订单状态查看。
4. 明确 Web 管理后台继续保留，小程序不承载运营后台。
5. 增加关键操作审计：司机审核、提现处理、退款处理、核销、参数调整。
6. 为客服和运营准备排障字段：本地订单号、供应商订单号、票号、支付流水号、请求时间、错误原因。

验收标准：

- 司机能在小程序中完成推广和提现申请的首版流程。
- 运营可以通过 Web 后台处理首版核心订单和司机事项。
- 资金、身份、订单状态相关操作有可追溯记录。

### 阶段 6：后端安全与一致性加固

目标：在真实接入前，把当前后端中会导致交易错乱或越权的基础问题修掉。

任务：

1. 将环岛游锁票、出票、退票、改签、解锁等交易类接口从公开路由移入用户或后台鉴权范围，并校验订单归属。
2. 核销改为条件更新：`WHERE id = ? AND status = 'valid'`，通过 affected rows 防止重复核销。
3. 支付和出票回调处理改为幂等，订单号、支付流水号、供应商订单号需要唯一约束或幂等键保护。
4. 未支付订单增加过期取消和释放锁定资源的机制。
5. 提现创建纳入事务，`pending`、`approved`、`transferred` 状态都应占用可提现余额。
6. 金额处理从 `float64` 逐步迁移到整数分或 decimal 类型库；至少保证支付、退款、佣金、提现路径不直接做浮点相等判断。
7. 证件号、收款账号等敏感字段增加加密存储和脱敏输出策略。
8. 生产环境启动时禁止默认 `JWT_SECRET`、默认支付密钥、空关键供应商凭据和宽松 CORS。

验收标准：

- 未登录或非订单所属用户不能触发交易状态变化。
- 重复点击、重复回调、重复核销、重复提现不会造成重复出票、重复入账或余额透支。
- 开发环境仍可使用 Mock/沙箱配置，生产环境不能带默认密钥启动。

### 阶段 7：真实接入与生产上线准备

目标：小程序前端重构和业务功能完成后，再接入真实外部服务，并把系统推进到可上线状态。

任务：

1. 接入微信登录和手机号授权，替换开发期手机号模拟登录。
2. 接入短信验证码或等价身份校验能力，用于必要的手机号验证和风控场景。
3. 接入微信支付：预下单、支付参数签名、回调验签、查单、关单、退款、退款回调、对账。
4. 接入环岛游供应商正式账号，完成锁票、出票、退票、改签、核销通知、余额查询和状态对账联调。
5. 完成生产 CORS、HTTPS、Nginx、数据库账号、密钥管理、备份恢复、日志脱敏和监控告警。
6. 建立数据库迁移版本管理，修正 Docker 初始化脚本与迁移脚本的完整性。
7. 做发布演练和回滚演练，编写上线 Runbook。
8. 提交微信小程序审核，按审核反馈修正页面、协议、支付和数据使用说明。

验收标准：

- 真实微信登录、真实支付、真实供应商出票、退款和核销通知全部通过联调。
- 生产环境具备可重复部署、备份恢复、监控告警和回滚方案。
- 小程序审核通过后，可按 Runbook 发布首版。

## 建议的最短上线路径

如果目标是先把项目改造成小程序，最短路径不应从真实支付或后端生产加固开始，而应先完成前端重构：

1. 用 `uni-app + Vue 3 + TypeScript` 新建小程序工程。
2. 迁移首页、环岛游订票、订单/票券、司机端、我的这几组页面。
3. 用 Mock 数据和现有后端公开接口跑通页面展示和跳转。
4. 前端重构验收后，再开发用户端交易功能和司机端功能。
5. 功能完成后，再集中做后端安全一致性、真实微信登录、微信支付、供应商正式接入和生产上线准备。

不建议第一步同时做真实登录、真实支付、正式供应商联调、完整退款资金流、提现真实打款和全量后台管理。它们会让“前端重构”变成无边界的全系统重做。
