# uni-app 迁移前端视觉还原修复计划

## /goal 提示词：uni-app 迁移前端视觉还原修复

将下面整份 Markdown 作为 `/goal` 输入使用。目标是修正当前 `miniprogram/` 迁移结果和原静态 HTML 原型之间的明显视觉偏差。原静态 HTML 浏览器截图是视觉真相源，迁移后的 uni-app H5 页面必须在相同视口、相同页面状态、相同内容场景下与原版截图对比到“普通人肉眼看不出明显区别”，才算该页面验收通过。任何阶段只要浏览器截图仍能明显看出迁移版与原版不同，就不得提交完成，也不得进入下一阶段。

## 背景

- [已验证] 当前迁移后的首页与原 `index.html` 首页视觉差异明显：布局、首屏 hero、插画/背景、色彩、卡片风格、底部导航、品牌区和整体质感均不一致。
- [已验证] 用户明确指出：迁移目标不是做一个“功能可运行但视觉另起炉灶”的小程序壳，而是要尽量还原原静态页面视觉。
- [已验证] 当前仓库同时存在原静态 HTML 和 `miniprogram/` uni-app 工程；原静态 HTML 应作为迁移视觉基线，而不是仅作为内容参考。
- [已验证] 旧静态页面本地服务端口约定为 `8000`，例如 `http://127.0.0.1:8000/index.html`。
- [已验证] 迁移后的 uni-app H5 本地服务端口约定为 `5174`，例如 `http://127.0.0.1:5174/#/pages/home/index`。
- [已验证] 之前的验收过度依赖构建、测试、包体和功能闭环，缺少严格的“原版 vs 迁移版”浏览器截图视觉对比门禁。
- [待验证] 每个已迁移页面的原版对应关系、首屏状态、滚动位置、数据场景、视口尺寸和截图基线需要在 Phase 1 中重新建立。

## 核心目标

- 把视觉还原设为最高优先级：迁移后的页面必须以原静态 HTML 截图为准，不允许随意改版、降级、简化或替换视觉语言。
- 为每个迁移页面建立原版浏览器截图基线和迁移版浏览器截图对比。
- 修复首页、我的、订单、票券、环岛游订票、司机端等已迁移页面的明显视觉偏差。
- 每个页面必须通过同视口浏览器截图对比；只有“迁移后和原版看不出明显区别”才算通过。
- 原版静态页面必须从 `8000` 端口截图；迁移版 uni-app H5 必须从 `5174` 端口截图。
- 保留现有 uni-app、Mock/API adapter、测试、包体和平台兼容能力，不用视觉修复破坏已有功能。
- 建立可复用视觉验收机制，后续页面迁移必须先截图对比再提交。

## 非目标

- 不在本目标内新增真实微信登录、微信支付、短信、供应商正式接入或后端业务能力。
- 不以“更现代”“更小程序化”“更清爽”为理由重设计页面。
- 不用功能可用、测试通过、包体合格替代视觉还原。
- 不用用户提供的截图作为唯一验收证据；必须由本地浏览器自动打开原版和迁移版页面并截图。
- 不要求像素级 100% 完全一致，但任何普通用户一眼能看出的布局、配色、素材、层级、间距、字体、卡片、导航差异都必须修复。

## 总体原则

- 原静态 HTML 浏览器截图是视觉真相源，迁移页必须向它靠拢。
- 端口约定是硬规则：原版静态页 `8000`，迁移版 uni-app H5 `5174`。
- 先建立截图基线，再改代码；没有原版截图基线不得开始视觉修复。
- 每个页面必须同视口、同滚动位置、同业务状态、同 Mock 数据场景截图对比。
- 每次视觉改动后必须重新截图，不接受“看代码应该差不多”。
- 浏览器截图必须保存到 `miniprogram/docs/validation/visual-parity/phase-N/`。
- 每个阶段都必须包含对比图或对比记录：原版截图、迁移版截图、差异说明、修复后截图、验收结论。
- 如果截图中仍能明显看出迁移版和原版不同，该阶段必须继续修复，不得标记 completed。
- 保持 API adapter、Mock 场景、平台禁用 API 扫描和包体预算；视觉修复不能引入 `window`、`document`、`localStorage`、`fetch`、外链脚本或不兼容小程序的 DOM 依赖。
- 每个阶段保持一个清晰主提交；提交前必须完成截图视觉验收和五类专项 review。
- 涉及代码 symbol 编辑前按仓库规则执行影响分析；提交前执行变更检测。

## 强制阶段门禁

1. 执行任何 Phase 前，先阅读本 goal 文档和当前 Phase，明确视觉真相源、页面范围、原版入口和迁移版入口。
2. 为当前 Phase 创建执行级 plan，列出页面、原版 URL、迁移版 URL、视口、截图文件、差异项、预计修改文件、测试命令和回滚边界。
3. 执行级 plan 必须经过至少两轮三角色对抗式优化：Builder reviewer、Skeptic reviewer、Verifier reviewer。
4. 若第二轮后仍有 P0/P1 缺陷，继续优化直到无 P0/P1 缺陷，或报告 blocker。
5. 只有优化后的 Phase plan 可以进入实现。
6. 实现前必须确认原版静态页服务运行在 `http://127.0.0.1:8000/`；未确认不得截图。
7. 实现前必须确认迁移版 uni-app H5 服务运行在 `http://127.0.0.1:5174/`；未确认不得截图。
8. 实现前必须采集原版浏览器截图基线；未生成原版截图不得开始改迁移页。
9. 每次完成视觉修复后，必须启动真实本地 H5 页面并采集迁移版浏览器截图。
10. 每个页面必须保存至少以下证据：
    - 原版截图
    - 修复前迁移版截图
    - 修复后迁移版截图
    - 差异清单
    - 验收结论
11. 视觉验收必须由浏览器截图对比完成，不能由用户截图、口头确认、代码审查或构建成功替代。
12. 验收结论只能是：
    - `通过：迁移版与原版截图肉眼无明显区别`
    - `不通过：仍存在明显视觉差异`
13. 只要任一页面为“不通过”，当前 Phase 不得提交、不允许进入下一 Phase。
14. 完成当前 Phase 后运行阶段指定的自动化测试、类型检查、H5 构建、`mp-weixin` 构建、平台 API 扫描和包体检查。
15. 提交前并行执行五类专项 review：Security、Logic bug、Test coverage、Maintainability、Performance / concurrency。
16. 任一专项 review 发现问题后，必须修复、重跑测试和浏览器截图验收，然后重跑全部五类专项 review。
17. 只有五类提交前 review 都明确 `未发现问题` 后，才允许执行提交准备检查。
18. 提交准备检查必须包括 `git status`、完整 `git diff`、凭据/生成物/缓存/无关文件检查。
19. 每个 Phase 主提交必须遵守 `.claude/rules/git-submit.mdc`。
20. 提交后必须 review 最新 commit diff；若发现问题，创建修复 commit 并重复 post-commit review。
21. 只有 post-commit review 无问题后，才允许把当前 Phase todo 标记为 completed 并进入下一 Phase。

## Review 门禁细则

提交前五类专项 review 必须使用自包含 packet：用户目标、当前 Phase、相关 diff、测试结果、原版截图、迁移版截图、视觉对比结论、`mp-weixin` 构建证据、仓库约束。

- Security reviewer：检查鉴权绕过、XSS、敏感数据泄露、危险日志、凭据读取、外链脚本引入。
- Logic bug reviewer：检查页面状态流、Mock/API adapter、路由跳转、错误态、空态、业务字段映射是否被视觉修复破坏。
- Test coverage reviewer：检查高风险 mapper、Mock 场景、脱敏、adapter、视觉验收命令是否有覆盖。
- Maintainability reviewer：检查样式组织、组件边界、命名、重复、设计 token、资源归属和后续扩展成本。
- Performance/concurrency reviewer：检查图片体积、包体压力、首屏资源、并发请求、缓存、构建体积和小程序包体风险。

每个 reviewer 必须返回以下两类结果之一：

- `未发现问题`
- 或按严重程度排序的问题清单，包含具体文件/路径引用和建议修复方式。

## 浏览器截图验收门禁

这是本 goal 的最高优先级门禁。

- 必须同时启动或服务原静态 HTML 与迁移后的 uni-app H5。
- 原版截图只能从 `8000` 端口采集：
  - `http://127.0.0.1:8000/index.html`
  - `http://127.0.0.1:8000/ticket.html`
  - `http://127.0.0.1:8000/driver.html`
  - `http://127.0.0.1:8000/island-cruise.html`
  - `http://127.0.0.1:8000/island-cruise-booking.html`
- 迁移版截图只能从 `5174` 端口采集：
  - `http://127.0.0.1:5174/#/pages/home/index`
  - `http://127.0.0.1:5174/#/pages/profile/index`
  - `http://127.0.0.1:5174/#/pages/orders/index`
  - `http://127.0.0.1:5174/#/pages/ticket/index`
  - `http://127.0.0.1:5174/#/pages/island-cruise/index`
  - `http://127.0.0.1:5174/#/subpackages/driver/pages/home/index`
- 迁移版默认使用 `VITE_API_MODE=mock`，确保数据稳定。
- 每个页面至少覆盖：
  - 移动视口：`390x844`
  - 用户反馈视口附近：`519x927`
  - 必要时桌面视口：`1440x900`
- 截图必须固定：
  - 相同页面状态
  - 相同滚动位置
  - 相同数据场景
  - 相同 viewport
  - 相同等待时间
- 每个页面必须检查：
  - 首屏布局
  - hero / banner / 背景图
  - logo 与品牌信息
  - 字体大小与字重
  - 主色、辅助色、渐变、透明度
  - 卡片圆角、阴影、边框
  - 图标风格
  - 间距、栅格、左右留白
  - 底部导航
  - 按钮样式
  - 空态、错误态、加载态
  - 文本截断、重叠、滚动和 sticky/fixed 区域遮挡
- “看不出明显区别”的判定标准：
  - 普通用户不应一眼看出这是两个不同设计稿。
  - 不允许出现明显不同的背景图、卡片风格、导航结构、主题色、布局密度或视觉层级。
  - 允许小程序平台导致的极小字体渲染差异、1-2px 级别间距差异、系统字体抗锯齿差异。
  - 不允许用“功能一样”解释视觉差异。
- 可使用像素 diff 辅助判断，但最终必须保存人工视觉结论。像素 diff 通过不能替代肉眼检查；肉眼明显不同即不通过。
- 没有截图对比证据，不得提交。

## Phase 1：建立原版视觉基线与逐页差异清单

### Codex Goal

建立视觉还原工作的事实基础：为原静态 HTML 和当前迁移版页面采集同视口截图，明确页面映射、差异清单、视觉验收阈值和后续修复顺序。

### Scope

- 确认原版页面与迁移页面映射：
  - `index.html` -> `miniprogram/src/pages/home/index.vue`
  - `index.html` mine/orders 区域 -> 我的、订单
  - `ticket.html` 与环岛游出票页 -> 票券详情
  - `island-cruise.html` / `island-cruise-booking.html` -> 环岛游订票
  - `driver.html` -> 司机端
- 启动原版静态页面本地服务，固定端口 `8000`。
- 启动迁移版 uni-app H5 mock 服务，固定端口 `5174`。
- 采集每组页面的原版截图、迁移版修复前截图。
- 输出差异清单，按 P0/P1/P2 排序。
- P0：普通用户一眼能看出页面完全不是同一套设计。
- P1：布局、主色、卡片、导航、关键视觉资产明显不一致。
- P2：局部间距、字体、阴影、图标细节偏差。
- 建立 `miniprogram/docs/validation/visual-parity/phase-1/` 证据目录。
- 不在本 Phase 大规模改 UI，只允许修复截图脚本、服务命令和文档。

### Acceptance Criteria

- 原版 `8000` 和迁移版 `5174` 本地服务均可打开。
- 每个迁移页面都有对应原版截图基线和迁移版修复前截图。
- 差异清单必须明确指出首页当前视觉不通过，且列出首屏 hero、背景图、卡片、导航、色彩等明显差异。
- 视觉验收标准写入文档，后续 Phase 必须引用。
- 自动化命令记录完整，包括原版服务命令、迁移版服务命令、截图命令。
- `typecheck`、`test`、`build:h5`、`build:mp-weixin`、`lint:platform`、`check:size` 保持通过。
- 提交前完成五类专项 review；全部 `未发现问题` 后才允许提交。
- 提交后再次 review 最新 commit；review 无问题后才允许进入 Phase 2。

## Phase 2：首页首屏与首页主体视觉还原

### Codex Goal

优先修复用户已指出的首页问题：迁移版首页必须对齐原 `index.html` 首页视觉，尤其是首屏 hero、品牌区、搜索框、热门目的地、推荐卡片和底部导航。

### Scope

- 以 Phase 1 采集的 `index.html` 首页截图为真相源。
- 修复迁移版首页：
  - 顶部 logo / 品牌信息
  - 搜索框位置和样式
  - 大幅插画/背景图
  - “您好～珠海！”或原版等效首屏文案
  - 主推区域“珠海湾区漫游”
  - 热门目的地卡片布局、图标、颜色、圆角、阴影
  - 船票/商品推荐卡片
  - 底部导航样式、图标、选中态
- 必须复用或迁移原版视觉资产；如原版资产不可直接用于小程序包，必须使用压缩后本地资源或明确记录替代原因。
- 不允许把当前简化版 hero 当作可接受迁移结果。
- 不改动无关后端、Docker 或旧静态 HTML。
- 修复后重新采集首页原版与迁移版截图。

### Acceptance Criteria

- 首页在 `390x844`、`519x927` 视口下，迁移版与原版截图肉眼无明显区别。
- 首页首屏不能再出现“明显是另一套设计”的情况。
- hero 背景、品牌区、搜索框、热门目的地卡片、底部导航必须视觉对齐。
- 截图验收文档必须包含：
  - 原版首页截图
  - 修复前迁移版首页截图
  - 修复后迁移版首页截图
  - 差异清单关闭情况
  - `通过：迁移版与原版截图肉眼无明显区别`
- 如果任何 reviewer 或执行者从截图中仍能明显看出迁移版与原版不同，本 Phase 不得完成。
- `typecheck`、`test`、`build:h5`、`build:mp-weixin`、`lint:platform`、`check:size` 全部通过。
- 提交前完成五类专项 review；全部 `未发现问题` 后才允许提交。
- 提交后再次 review 最新 commit；review 无问题后才允许进入 Phase 3。

## Phase 3：我的、订单、票券基础页视觉还原

### Codex Goal

修复主包剩余基础页面的视觉偏差，使“我的”、订单列表、票券详情在原版对应状态下与静态 HTML 原型截图看不出明显区别。

### Scope

- 以 `index.html` mine/orders 区域和 `ticket.html` 为真相源。
- 修复：
  - 我的页用户信息区、入口卡片、服务入口、视觉层级。
  - 订单页筛选、订单卡片、状态标签、按钮、空态。
  - 票券详情页票券视觉、二维码/票码 fallback、说明区、底部操作。
- 保持现有 Mock 场景和 adapter 不被破坏。
- 空态、未登录、失败态也必须有原版或合理迁移视觉基线；如果原版缺少对应状态，必须以原版视觉语言扩展，而不是另起风格。
- 每个页面必须完成截图对比。

### Acceptance Criteria

- 我的、订单、票券详情在移动视口下与原版对应区域/页面肉眼无明显区别。
- 状态标签、按钮、卡片、间距、字体、底部导航与原版视觉语言一致。
- 票码、手机号、证件号继续脱敏。
- 所有截图证据保存到 `visual-parity/phase-3/`。
- 任何页面截图仍明显像另一套设计，则本 Phase 不通过。
- `typecheck`、`test`、`build:h5`、`build:mp-weixin`、`lint:platform`、`check:size` 全部通过。
- 提交前完成五类专项 review；全部 `未发现问题` 后才允许提交。
- 提交后再次 review 最新 commit；review 无问题后才允许进入 Phase 4。

## Phase 4：环岛游订票与司机端视觉还原

## 总体最终验收标准

- 原版静态页固定从 `8000` 端口采集截图，迁移版 uni-app H5 固定从 `5174` 端口采集截图。
- 迁移后的 uni-app H5 页面与原静态 HTML 页面在同视口浏览器截图下肉眼无明显区别。
- 首页必须首先通过视觉验收；首页不通过，不允许继续后续页面。
- 所有已迁移页面必须有原版截图、修复前截图、修复后截图和最终通过结论。
- 构建、测试、包体、平台 API 扫描全部通过，但这些只作为基础质量门槛，不能替代视觉还原验收。
- 没有任何页面被“重新设计”“风格统一”“小程序化”作为偏离原版的理由。
- Mock/API adapter、脱敏、storage、导航、平台兼容能力保持有效。
- 小程序包体仍满足主包 `< 2 MB`、分包 `< 2 MB`、总包 `< 8 MB`。
- 现有静态 HTML 不被删除，直到视觉验收完成并经用户明确要求归档或移除。
- 最终状态必须明确：视觉还原通过、build-validated 通过；若缺微信开发者工具运行时截图，仍需记录为正式提测前 blocker。
